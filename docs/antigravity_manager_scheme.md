# Antigravity Manager: 原生 macOS 客户端集成方案

## 1. 背景与目标
目前 Antigravity 的解锁与启动依赖于一系列离散的 Shell 脚本（解锁、数据迁移、注入启动），操作链路长且容易出错。本方案旨在开发一个原生的 macOS 菜单栏应用（Antigravity Manager），将所有后台逻辑封装进一个直观的 GUI 界面中，实现“一键即达”的使用体验。

## 2. 核心流程设计

### 2.1 自动化初始化 (One-Click Setup)
管理器启动后会检测环境状态，若为首次运行或检测到原版 App 更新，将引导用户执行初始化：
1. **自动寻径**：定位 `/Applications/Antigravity.app`。
2. **静默解锁**：在后台执行重签名逻辑（替代 `robust_unlock.sh`），并弹出一次性系统授权框。
3. **数据同步**：自动迁移原版沙盒数据到本地配置目录（替代 `migrate_data.sh`），重置 TCC 权限。

### 2.2 运行管理 (Runtime Management)
通过菜单栏图标提供常驻控制：
- **状态感知**：实时显示代理进程是否存活、Dylib 是否成功注入。
- **一键启停**：替代 `run_unlocked.sh`，以正确的环境变量（`DYLD_INSERT_LIBRARIES`）拉起 Electron 进程。
- **自动退出**：当管理器关闭时，可选择同步关闭代理进程，防止资源残留。

## 3. 功能特性 (Features)

| 功能模块 | 说明 |
| :--- | :--- |
| **状态栏图标** | 使用 SF Symbols (如 `bolt.shield.fill`) 动态显示运行状态。 |
| **快捷菜单** | 包含：启动/停止、查看实时日志、编辑配置文件、重新执行解锁。 |
| **日志监控器** | 内置简易日志查看器，实时轮询 `/tmp/antigravity_proxy.*.log`。 |
| **配置编辑器** | 提供简单的 UI 或快捷打开 `config.json` 进行代理参数修改。 |
| **自动更新检测** | 监控原版 App 的镜像，若发现原版已更新，提示用户重新执行一键解锁。 |

## 4. 技术实现细节

- **开发语言**：Swift 6.0 + AppKit (轻量级，无需庞大的 runtime)。
- **进程控制**：使用 `Foundation.Process` (NSTask) 管理后台进程，捕获 stdout/stderr。
- **UI 模式**：`LSUIElement = true` (仅菜单栏显示，不占用 Dock 栏)。
- **权限处理**：通过 `Process` 调用系统 `codesign` 和 `tccutil` 工具。

## 5. 潜在问题与应对方案

### 5.1 钥匙串 (Keychain) 重新授权
- **问题**：重签名后，macOS 识别为“新应用”，会丢失原有的登录 Token。
- **对策**：在方案文档和 UI 中明确提示：**“首次运行需重新登录，之后将永久保存。”**

### 5.2 系统隐私弹窗 (TCC)
- **问题**：迁移数据并重置权限后，系统会再次询问位置、麦克风等权限。
- **对策**：这属于 macOS 标准安全行为，管理器会在首次点击“启动”前弹出 Dialog 预警，降低用户焦虑。

### 5.3 SIP 与环境变量限制
- **问题**：`DYLD_INSERT_LIBRARIES` 在特定受限环境下可能失效。
- **对策**：管理器将直接执行 `Contents/MacOS/Electron` 二进制文件，绕过 `open` 命令带来的环境变量清洗。

## 6. 结论
通过原生 App 化，我们将 Antigravity 从一个“脚本工具”转变为一个“系统服务”。这不仅提升了稳定性，也为后续增加流量统计、多配置切换等高级功能打下了基础。
